# VibeFlow 状态与阻断系统全景分析

> 本文档梳理 VibeFlow 当前所有状态转换逻辑、阻断行为、以及使用中遇到的问题，最终给出终极形态的设计方向。

## 一、Daily State Machine

### 1.1 五个状态

```
                    COMPLETE_AIRLOCK
  ┌────────┐  ─────────────────────>  ┌──────────┐
  │ LOCKED │                          │ PLANNING │ <──── COMPLETE_REST / ABORT_POMODORO
  └────────┘                          └──────────┘
       ^                                 |     ^
       |  DAILY_RESET                    |     |
       |  (04:00, 任意状态)              |     |
       |                    START_POMODORO     |
       |                                 |     |
       |                                 v     |
       |                             ┌───────┐ |
       +<────────────────────────────│ FOCUS │─+── COMPLETE_POMODORO ──> REST
       |                             └───────┘                          |
       |                                 ^                              |
       |                                 |                              v
       |                        START_POMODORO                    ┌──────────┐
       |                                 |                        │   REST   │
       |                           ┌───────────┐                 └──────────┘
       +<──────────────────────────│ OVER_REST │ <───── ENTER_OVER_REST (from REST/PLANNING)
                                   └───────────┘ <───── COMPLETE_POMODORO (if isInOverRest)
```

### 1.2 转换表

| 源状态 | 事件 | 目标状态 | 守卫 | 说明 |
|--------|------|---------|------|------|
| LOCKED | COMPLETE_AIRLOCK | PLANNING | top3 ≤ 3 tasks | 完成 airlock |
| PLANNING | START_POMODORO | FOCUS | count < cap | 开始番茄钟 |
| PLANNING | ENTER_OVER_REST | OVER_REST | — | 服务器检测超时 |
| FOCUS | COMPLETE_POMODORO | REST | !isInOverRest | 番茄钟完成 |
| FOCUS | COMPLETE_POMODORO | OVER_REST | isInOverRest | 番茄钟完成但已超时 |
| FOCUS | ABORT_POMODORO | PLANNING | — | 中止 |
| REST | COMPLETE_REST | PLANNING | — | 休息结束 |
| REST | ENTER_OVER_REST | OVER_REST | — | 休息超时 |
| REST | OVERRIDE_CAP | PLANNING | — | 覆盖日上限 |
| OVER_REST | START_POMODORO | FOCUS | count < cap | 从超时中恢复 |
| *任意* | DAILY_RESET | LOCKED | — | 每日 04:00 |

### 1.3 每个状态下用户能做什么

| 状态 | 用户可以做 | 用户不能做 |
|------|-----------|-----------|
| **LOCKED** | 完成 airlock（选择 Top 3 任务） | 其他所有操作 |
| **PLANNING** | 开始番茄钟、管理任务、AI Chat | — |
| **FOCUS** | 完成/中止番茄钟、切换任务、AI Chat（受限） | 修改设置 |
| **REST** | 完成休息、覆盖日上限、AI Chat | 修改敏感设置 |
| **OVER_REST** | 开始新番茄钟 | 大部分操作 |

## 二、阻断系统 — 三个客户端的现状

### 2.1 阻断触发源

系统中有 **5 种阻断/干预触发源**，分布在不同客户端：

| 触发源 | 含义 | 判定方 | 影响的客户端 |
|--------|------|--------|-------------|
| **focus** | 番茄钟进行中 | 服务器 (FOCUS state) | iOS: shield 分心App; 浏览器: 关闭黑名单标签页 |
| **over_rest** | 休息超时 | 服务器 (overRestService 计算) | iOS: shield 分心App; 桌面: 关闭分心App+置顶; 浏览器: 重定向 |
| **sleep** | 睡眠时间窗口 | 服务器 (sleepTimeService) | iOS: shield 分心App; 桌面: 关闭分心App |
| **idle** | 工作时间内空闲超阈值 | 桌面端本地 (FocusEnforcer) | 桌面: 置顶窗口+通知+关闭分心App |
| **locked** | LOCKED 状态 | 服务器 | 浏览器: 重定向所有标签页到锁屏页 |

### 2.2 各客户端阻断行为对比

#### 桌面端 (Electron)

三个独立 Enforcer：

| 模块 | 触发条件 | 动作 | 轮询频率 |
|------|---------|------|---------|
| FocusEnforcer | 工作时间 + 无番茄钟 + 空闲超阈值 | 置顶窗口 + 通知 + 关闭/隐藏分心App | 10 秒 |
| OverRestEnforcer | policy.overRest.shouldTriggerActions | 关闭分心App + 30秒置顶窗口 | policy 更新时 |
| SleepEnforcer | policy.sleepTime.isCurrentlyActive | 关闭分心App (force_quit) | 10 秒 |

执行方式：AppleScript → force_quit（严格模式）或 hide_window（温和模式）

#### iOS 端 (React Native)

单一 blockingService 监听 Zustand Store：

```
evaluateBlockingReason():
  1. activePomodoro?.status === 'active'  → 'focus'     (最高优先级)
  2. policy?.overRest?.isOverRest         → 'over_rest'
  3. policy?.sleepTime 启用且活跃且未贪睡  → 'sleep'
  4. else                                 → null
```

执行方式：ManagedSettingsStore.shield.applicationCategories = .all()（全品类阻断）

#### 浏览器扩展 (Chrome Extension)

```
checkUrlPolicy():
  1. LOCKED/OVER_REST 状态 → 重定向所有非 Dashboard 标签页
  2. 娱乐站点 + 非娱乐模式 → 阻断
  3. 工作时间 + 无番茄钟 + 黑名单 URL → 严格关闭/温和提醒
  4. 未知 URL + 工作时间 + 无番茄钟 → 软阻断
```

### 2.3 每种状态下的阻断行为矩阵（现状）

| 状态 | 桌面-分心App | 桌面-工作App | iOS-分心App | 浏览器-黑名单 | 浏览器-娱乐站 |
|------|-------------|-------------|------------|-------------|-------------|
| **LOCKED** | ⚪ 无 | ⚪ 无 | ⚪ 无 | 🔴 全部重定向 | 🔴 阻断 |
| **PLANNING** | ⚠️ 空闲干预 | ⚪ 无 | ⚪ 无 | ⚠️ 工作时间内阻断 | 🔴 阻断 |
| **FOCUS** | ⚪ 无¹ | ⚪ 无 | 🔴 全品类shield | 🔴 阻断 | 🔴 阻断 |
| **REST** | ⚪ 无 | 🔵 可选阻断 | ⚪ 无 | ⚪ 无²  | 🔴 阻断³ |
| **OVER_REST** | 🔴 关闭+置顶 | ⚪ 无 | 🔴 全品类shield | 🔴 全部重定向 | 🔴 阻断 |
| **SLEEP** | 🔴 关闭 | ⚪ 无 | 🔴 全品类shield⁴ | 🔴 阻断 | 🔴 阻断 |

注释：
1. 桌面 FOCUS 不主动关闭分心App — 用户在电脑上工作，浏览器扩展已处理网页
2. REST 时浏览器扩展不阻断黑名单 — 但有 `shouldBlockEnhanced()` 可能在工作时间内仍阻断，逻辑有歧义
3. 娱乐站点在任何状态下都阻断（除非开启娱乐模式）— 娱乐模式仅在非工作时间可用
4. iOS sleep 阻断依赖 WebSocket 连接 + App 存活 — 不可靠

## 三、OVER_REST — 当前系统中最混乱的部分

### 3.1 两条进入路径

```
路径 A — 显式（XState 事件）:
  服务器代码调用 → ENTER_OVER_REST → 数据库写入 state = 'over_rest'

路径 B — 隐式（查询时覆盖）:
  数据库存储 state = 'rest' 或 'planning'
  查询时调用 overRestService.checkOverRestStatus()
  如果超时 → API 返回 state = 'over_rest'（但数据库没变）
```

出现隐式覆盖的位置：
- `daily-state.service.ts` → `getTodayWithProgress()` — 查询时
- `socket.ts` → `sendStateSnapshotToSocket()` — 发送快照时
- `socket.ts` → 30 秒定期广播 — 定期轮询时

**问题：** 数据库中的 `state` 字段不一定反映真实状态，导致调试困难。

### 3.2 Over-Rest 检测算法

```
overRestService.checkOverRestStatus(userId):
  1. 获取 shortRestDuration (默认 5min), gracePeriod (默认 5min)
  2. 不在工作时间且无临时专注会话 → NOT over-rest
  3. 有 IN_PROGRESS 番茄钟 → NOT over-rest
  4. 找最近 COMPLETED 番茄钟 endTime
  5. restDuration = now - endTime
  6. isOverRest = restDuration > shortRestDuration
  7. shouldTriggerActions = isOverRest && (restDuration - shortRestDuration) >= gracePeriod

时间线:
  |── 番茄钟结束 ──|── shortRest 5min ──|── gracePeriod 5min ──|
  |   正常休息      |  isOverRest=true    | shouldTriggerActions  |
  |                 |  （仅标记）          | = true → 执行动作     |
```

### 3.3 iOS vs 桌面端阈值不一致

- **iOS** 检查 `policy.overRest.isOverRest` → 5 分钟后即阻断
- **桌面端** 通过 OverRestEnforcer 检查 policy → 内部使用 `shouldTriggerActions` → 10 分钟后才阻断
- **结果：iOS 比桌面端早 5 分钟触发阻断**

## 四、实际使用中的问题分析

### 问题 1: Airlock（空气开关）被频繁 skip

**现象：** 每天的 airlock 很少认真做，大多直接跳过。

**根本原因分析：**

| 原因 | 说明 |
|------|------|
| **仪式感 vs 摩擦** | Airlock 设计初衷是"晨间仪式"，但实际体验是"每天早上一个拦路虎" |
| **Top 3 选择压力** | 必须选 3 个任务才能解锁，但有时候只有 1-2 个明确任务，或者还没想好 |
| **时机不对** | 04:00 重置后进入 LOCKED，但用户真正坐下来工作可能是 9:00。这 5 小时内 App 完全不可用（浏览器扩展重定向所有页面） |
| **缺乏引导** | 没有 AI 参与，就是一个冷冰冰的选择界面 |
| **没有回报感** | 完成 airlock 后直接进入 PLANNING，没有正反馈 |

**当前系统的处理：**
- `skipAirlockForNewUser()` 只对"没有任务的新用户"有效
- 普通用户如果想 skip，没有官方路径 → 但实际上用户在 Web 端可以绕过

### 问题 2: 工作节奏不稳定

**现象：** 工作时间内不按番茄钟节奏走，时而专注时而摸鱼，番茄钟形同虚设。

**根本原因分析：**

| 原因 | 说明 |
|------|------|
| **启动摩擦大** | 每次开始番茄钟需要：选任务 → 点开始 → 等倒计时。如果任务列表不清晰，光选任务就要 1 分钟 |
| **中断代价高** | 番茄钟中如果需要处理紧急事务，只能"中止"整个番茄钟，25 分钟白费了的感觉 |
| **休息被跳过** | REST 阶段没有强制执行（桌面端不阻断分心 App），用户经常直接跳过休息继续干活 |
| **Over-rest 检测滞后** | shortRestDuration (5min) + gracePeriod (5min) = 10 分钟后才开始动作，用户已经刷了 10 分钟手机了 |
| **PLANNING 状态无约束** | PLANNING 里可以无限期停留，没有任何推力让用户开始番茄钟 |
| **特殊情况无法处理** | 开会、突然被打断、需要临时处理事务 — 都不符合"25 分钟纯专注"的模型 |

**当前系统的应对：**
- Ad-hoc Focus Session：可以不通过番茄钟直接启动专注模式 — 部分解决
- FocusEnforcer 空闲检测：PLANNING 时空闲超阈值会干预 — 但仅桌面端
- Over-rest escalation：逐步加强提醒 — 但起步太晚

### 问题 3: 摸鱼检测不到位

**现象：** 在 Chrome 里摸鱼、在手机上摸鱼，系统察觉不到或反应太慢。

**Chrome 摸鱼的现状：**

| 场景 | 系统行为 | 问题 |
|------|---------|------|
| FOCUS + 黑名单 URL | 浏览器扩展关闭标签页 | ✅ 有效 |
| FOCUS + 非黑名单 URL | 无动作 | ❌ 用户在非黑名单网站（如知乎、V2EX）摸鱼 |
| PLANNING + 黑名单 URL | 工作时间内阻断 | ⚠️ 取决于是否配置了黑名单 |
| PLANNING + 任意 URL | 软阻断（温和提醒） | ⚠️ 用户可以轻松关闭提醒 |
| REST + 任意 URL | 不阻断 | ✅ 合理，在休息 |
| 非工作时间 | 不阻断 | ✅ 合理 |

**手机摸鱼的现状：**

| 场景 | 系统行为 | 问题 |
|------|---------|------|
| FOCUS + 分心 App | 全品类 shield | ❌ 过度 — 连电话都接不了，所以用户根本不会开启 |
| FOCUS + 分心 App（未开启 shield） | 无动作 | ❌ 手机完全不受控制 |
| PLANNING + 任意 App | 无动作 | ❌ 手机完全不受控制 |

**核心矛盾：**
- Chrome 摸鱼：黑名单机制太静态，无法覆盖所有摸鱼场景
- 手机摸鱼：全品类阻断体验太差，用户宁可不开

### 问题 4: "已经在工作了"但没启动番茄钟

**现象：** 用户直接开始写代码/做事，忘了或懒得启动番茄钟。系统认为用户在"摸鱼"。

**场景：**
- 早上到工位，直接打开 IDE 开始写代码，完全没想起来开番茄钟
- 临时被同事叫去 review 代码，回来后继续写，从头到尾没有番茄钟
- 处理紧急 bug，争分夺秒，哪有心思去点"开始番茄钟"

**问题本质：** 番茄工作法的价值在于**有节奏的工作-休息循环**，而不是"必须点一个按钮才算开始工作"。用户实际上在高效工作，但系统认为他在 PLANNING 状态空闲，甚至可能触发 FocusEnforcer 干预——这反而打断了用户。

**理想行为：** 系统能感知到"用户正在工作"（IDE 活跃、键盘敲击、代码提交），自动进入保护模式，并在合适的时机建议休息。

### 问题 5: 阻断功能在社交场景中的尴尬

**现象：** 在非番茄钟的工作时间，如果需要给同事展示内容、在会议中投屏，阻断功能可能会造成尴尬。

**场景：**
- 正在投屏给同事看方案，浏览器扩展突然弹出"你在摸鱼"的提醒页面
- 在会议中分享屏幕，切到某个标签页被重定向到 VibeFlow 锁屏页
- 同事来工位看问题，电脑上弹出"空闲超时，请开始工作"的通知

**问题本质：** 阻断系统不理解**社交上下文**。工作不只是一个人对着电脑，还包括协作、讨论、展示。在这些场景下，阻断功能应该自动静默。

**可能的解决方向：**
- 检测系统是否在投屏/屏幕共享状态 → 自动暂停阻断
- 用户可以快速进入"展示模式"（一键暂停所有阻断 30 分钟）
- AI 识别日历中的会议 → 自动调整阻断策略

### 问题 6: 规则死板，缺乏"和系统对话"的能力

**现象：** 基于规则的系统无法处理特殊情况，用户要么遵守要么绕过，没有中间地带。

**场景：**
- 晚上 11 点，睡眠阻断生效了，但用户有一个 deadline 明早要交 → 当前只能去系统设置手动关闭 Screen Time
- 今天身体不舒服，不想做 5 个番茄钟 → 当前没有告诉系统"今天状态不好"的方式
- 下午有个紧急会议，需要临时调整工作计划 → 只能手动改 Top 3

**理想行为：用户可以和 AI 对话来协商规则。**

```
用户: "今晚有个紧急的 deadline，需要加班到凌晨 1 点"
AI:   "了解。我帮你把睡眠阻断推迟到 01:00，但 01:00 之后会强制生效。
       另外提醒你，连续工作超过 2 小时建议休息 15 分钟。要我设个提醒吗？"

用户: "今天状态不太好，不想做太多"
AI:   "没关系。我把今天的目标调整为 2 个番茄钟，先做最重要的那个任务。
       如果后面状态好了，随时可以加。"

用户: "下午 3 点有个会议，大概 1 小时"
AI:   "好的，我在 15:00-16:00 暂停所有阻断和提醒。
       会议结束后我提醒你继续下午的任务。"
```

**关键原则：**
- AI 是"教练"而不是"看守"——它理解你的意图，帮你调整计划，而不是僵硬执行规则
- 用户需要保持诚实（这是一个前提假设——如果用户刻意欺骗 AI，系统无法也不应该强制阻止）
- AI 可以温和地质疑："你这周已经第三次推迟 deadline 了，要不要聊聊是什么在阻碍你？"
- 但最终决定权在用户手上

### 问题 7: 缺乏智能 — 不理解上下文，只知道规则

**现象：** 系统是"规则驱动"的，不理解用户的实际状态和意图。

| 场景 | 理想行为 | 现实 |
|------|---------|------|
| 用户在 PLANNING 已经 30 分钟了 | AI 主动问"需要帮你规划一下吗？" | 空闲干预弹窗（仅桌面端） |
| 用户反复中止番茄钟 | AI 问"今天状态不太好？要不要调整一下节奏？" | 无响应 |
| 用户在 REST 超时 5 分钟 | AI 温和提醒"休息差不多了" | Over-rest 要等 10 分钟后才开始动作 |
| 用户连续 3 个番茄钟做同一个任务 | AI 建议"要不要把这个任务拆小一点？" | chat-triggers 有 task_stuck 但需要 AI chat 启用 |
| 早上坐下来准备工作 | AI 根据昨天的完成情况建议今天的 Top 3 | Airlock 只是冷冰冰的选择界面 |
| 晚上有紧急 deadline | AI 调整阻断时间，同时设置健康提醒 | 只能手动去系统设置关闭 |

**已有但未充分利用的 AI 能力：**
- `chat-triggers-state.service.ts`：5 种状态触发器（planning、rest、over_rest、escalation、task_stuck）
- `chat-triggers-cron.service.ts`：定时触发（morning_greeting、midday_check、evening_summary）
- `chat-intent.service.ts`：意图分类 + 状态感知工具子集
- `ai-trigger.service.ts`：触发框架（冷却、静默时间、FOCUS 保护）
- `chat.service.ts`：完整的对话能力 + 28 个 MCP 工具 — **这是实现"和系统对话"的基础设施**

## 五、不合理之处总结

### 5.1 架构级问题

| 编号 | 问题 | 严重程度 | 说明 |
|------|------|---------|------|
| A1 | OVER_REST 双重身份（存储 vs 计算） | 🔴 高 | DB 可能存 `rest`，API 返回 `over_rest`。两条进入路径导致不一致 |
| A2 | iOS/桌面端 over-rest 阈值不一致 | 🟡 中 | iOS 用 `isOverRest`（5min），桌面端用 `shouldTriggerActions`（10min） |
| A3 | Sleep 不在状态机中 | 🟡 中 | Sleep 是独立时间窗口，与 Daily State 正交但无协调机制 |
| A4 | 浏览器扩展 REST 状态阻断逻辑歧义 | 🟡 中 | `shouldBlockEnhanced()` 在 REST+工作时间 时的行为不明确 |

### 5.2 产品体验问题

| 编号 | 问题 | 严重程度 | 说明 |
|------|------|---------|------|
| P1 | Airlock 摩擦太大 | 🔴 高 | 每天第一个动作是障碍，不是助力 |
| P2 | 番茄钟启动摩擦大 | 🔴 高 | 选任务→点开始，每次都是决策负担 |
| P3 | "已经在工作了"但没番茄钟 | 🔴 高 | 用户直接开始写代码，系统却认为在空闲 |
| P4 | PLANNING 状态无推力 | 🟡 中 | 可以无限期停留，没有递进式引导 |
| P5 | REST 状态无约束 | 🟡 中 | 桌面端休息时不阻断分心 App，用户跳过休息继续工作或摸鱼 |
| P6 | 全品类阻断不可用 | 🔴 高 | iOS `.all()` 导致连电话都接不了，用户不会启用 |
| P7 | 黑名单覆盖不全 | 🟡 中 | 静态黑名单无法覆盖所有摸鱼场景（知乎、V2EX、HN 等） |
| P8 | 社交场景尴尬 | 🟡 中 | 投屏/展示时弹出阻断提醒，在同事面前尴尬 |
| P9 | 中断恢复体验差 | 🟡 中 | 被打断后只能中止番茄钟，没有"暂停+恢复"机制 |
| P10 | 规则死板无法协商 | 🔴 高 | 特殊情况（加班、会议、身体不适）无法和系统沟通 |

### 5.3 智能化缺失

| 编号 | 问题 | 严重程度 | 说明 |
|------|------|---------|------|
| I1 | 规则驱动而非理解驱动 | 🔴 高 | 系统不理解用户正在做什么，只知道时间和状态 |
| I2 | 无法感知工作状态 | 🔴 高 | 用户在 IDE 里写代码，系统不知道；用户在 Chrome 摸鱼，系统也迟钝 |
| I3 | 无法处理特殊情况 | 🔴 高 | 加班、会议、紧急 deadline 等场景下，系统不能灵活调整 |
| I4 | AI 触发器未充分利用 | 🟡 中 | 5 种状态触发 + 定时触发已实现，但需要 AI Chat 完全就绪 |
| I5 | 无行为建模 | 🟡 中 | 不追踪用户的行为模式（每天什么时候状态最好、什么时候容易摸鱼） |
| I6 | 无适应性 | 🟡 中 | 规则是静态的，不根据用户实际表现调整 |

## 六、终极形态 — 从"规则引擎"到"AI 行为教练"

### 6.0 终极目标

> **在合适的时间，合适的状态，做合适的事情。**

这不是一个效率工具的目标，而是一个**符合个人价值观的生活方式**。VibeFlow 的终极形态不是"让你多做几个番茄钟"，而是帮你实现你自己定义的"好的一天"：

- 该工作的时候深度工作
- 该休息的时候真正休息
- 该睡觉的时候放下手机
- 遇到特殊情况能灵活调整，而不是被系统困住

系统应该**理解你的长期目标**（Goals/Projects），**感知你的当前状态**（在工作？在摸鱼？在开会？累了？），然后**给出最合适的下一步建议**。

### 6.1 设计哲学转变

```
现状:  规则驱动 → 被动响应 → 惩罚式干预
       "你违反规则了，我要阻断你"

终极:  AI 驱动 → 主动引导 → 赋能式助推
       "我理解你的状态，帮你找到最好的下一步"
```

**三层模型：规则 → AI → 用户自主**

```
┌─────────────────────────────────────────────────┐
│  第三层: 用户自主                                  │
│  用户可以和 AI 对话，协商调整任何规则               │
│  "今晚加班到 1 点" → AI 调整阻断 + 设健康提醒      │
│  前提: 用户保持诚实                                │
├─────────────────────────────────────────────────┤
│  第二层: AI 智能                                   │
│  理解上下文，在规则之上做灵活判断                    │
│  感知工作状态、识别社交场景、连接长期目标            │
│  在正确时机给正确提示                               │
├─────────────────────────────────────────────────┤
│  第一层: 规则引擎                                  │
│  状态机、阻断策略、时间窗口、黑白名单               │
│  确定性的、可预测的基础行为                         │
│  AI 和用户都在这层之上运作                          │
└─────────────────────────────────────────────────┘
```

规则是地基，AI 是教练，用户是运动员。教练理解运动员的目标和状态，在规则框架内做出最优指导；运动员可以和教练沟通调整训练计划。

**引入福格行为模型 (Fogg Behavior Model)：**

```
行为 = 动机 × 能力 × 提示

B = M × A × P
```

| 要素 | 含义 | VibeFlow 中的体现 |
|------|------|-------------------|
| **动机 (Motivation)** | 用户想要完成目标的意愿 | AI 理解用户的 Goals/Projects，在合适的时机连接"当前任务"和"长期目标" |
| **能力 (Ability)** | 完成行为的难度 | 降低启动番茄钟的摩擦、自动选择任务、缩短到"一句话就开始" |
| **提示 (Prompt)** | 触发行为的信号 | 在正确的时机给出正确的提示——不是弹窗，是 AI 对话中的自然建议 |

**关键洞察：** 现在的系统在动机不足时用惩罚（阻断）来补偿，但福格模型告诉我们——如果动机不足，应该降低难度和优化提示，而不是增加痛苦。阻断是最后的防线，不是常态手段。

### 6.2 Daily State Machine 终极版

#### 核心变更：Airlock 从"门卫"变成"AI 教练晨间对话"

```
现状：
  LOCKED ──(手动选3个任务)──> PLANNING ──(手动点开始)──> FOCUS

终极版：
  LOCKED ──(AI 自动建议 Top3，用户确认)──> READY
         ──(或用户直接说"开始工作")──> FOCUS
```

**具体改进：**

| 阶段 | 现状 | 终极 |
|------|------|------|
| **晨间** | 必须选 3 个任务才能解锁 | AI 主动推送建议（基于昨日进展、逾期任务、日历），用户说"可以"即进入 READY |
| **启动** | 手动选任务 + 点开始 | 对 AI 说"开始"或"做 XXX"，一句话启动番茄钟 |
| **专注中** | 被动等待完成 | AI 安静守护，只在被打断时给建议（"要不要暂停？还剩 8 分钟"） |
| **休息** | 被动倒计时 | AI 在 REST 进入时总结本轮，在 REST 即将结束时建议下一个任务 |
| **超时** | 10 分钟后弹窗 | AI 从第 3 分钟开始温和提醒，逐步升级，理解上下文 |

#### 新状态：READY（取代 PLANNING 的大部分职责）

```
LOCKED ──(AI辅助晨间规划)──> READY ──(说"开始"或AI建议)──> FOCUS
                                 ^                            |
                                 |                            v
                                 +<────── COMPLETE_REST ──── REST
```

**READY vs PLANNING 的区别：**
- PLANNING 是被动等待状态，用户什么都能做，也什么都不做
- READY 是"万事俱备，只欠东风"——Top 3 已确定，下一个任务已准备好，只需一个行动信号

#### Airlock 改造方案

| 方案 | 描述 | 摩擦 | 适合场景 |
|------|------|------|---------|
| **AI 建议确认** | AI 分析昨日进展+逾期任务，推送 Top 3 建议，用户一键确认 | 极低 | 常规工作日 |
| **语音启动** | "开始工作" → AI 自动选择最优先任务并启动番茄钟 | 极低 | 赶时间 |
| **延迟 Airlock** | 允许先进入 READY，在第一个番茄钟开始前完成 Top 3 选择 | 低 | 先做紧急任务 |
| **完全跳过** | AI 根据历史行为自动设置 Top 3（用户可事后调整） | 零 | 持续项目期间 |

### 6.3 阻断策略终极版

#### 统一阻断行为矩阵

**原则：**
1. FOCUS 时保护专注（阻断分心）
2. REST 时保护休息（可选阻断工作App）
3. OVER_REST 时推动回归（阻断分心）
4. SLEEP 时保护睡眠（阻断分心）
5. 非工作时间不干预（除了 SLEEP）

**iOS 端终极矩阵：**

| Daily State \ 时段 | 工作时间 | 非工作时间 | 睡眠时间 |
|--------------------|---------|-----------|---------|
| **LOCKED** | ⚪ 无 | ⚪ 无 | 🔴 分心App |
| **READY** | ⚪ 无 | ⚪ 无 | 🔴 分心App |
| **FOCUS** | 🔴 分心App | 🔴 分心App | 🔴 分心App |
| **REST** | ⚪ 无 | ⚪ 无 | 🔴 分心App |
| **OVER_REST** | 🔴 分心App | ⚪ 无 | 🔴 分心App |
| **SLEEP 覆盖** | — | — | 🔴 分心App（离线可用） |

**桌面端终极矩阵：**

| Daily State \ 时段 | 工作时间 | 非工作时间 | 睡眠时间 |
|--------------------|---------|-----------|---------|
| **LOCKED** | ⚪ 无 | ⚪ 无 | 🔴 分心App |
| **READY** | ⚠️ AI 引导开始工作 | ⚪ 无 | 🔴 分心App |
| **FOCUS** | ⚪ 无¹ | ⚪ 无 | ⚪ 无（正在专注） |
| **REST** | 🔵 可选工作App阻断 | ⚪ 无 | 🔴 分心App |
| **OVER_REST** | 🔴 分心App+置顶 | ⚪ 无 | 🔴 分心App |

¹ 桌面端 FOCUS 不主动关闭分心 App（用户在电脑上工作），由浏览器扩展处理网页

#### Over-Rest 统一为"计算态"

**决策：OVER_REST 不再写入数据库，统一为查询时计算。**

所有客户端统一使用 `shouldTriggerActions` 作为阻断触发条件：

```
统一时间线:
  |── 番茄钟结束 ──|── shortRest 5min ──|── gracePeriod 5min ──|
  |   REST (正常)   |  AI 温和提醒(新增) | shouldTriggerActions  |
  |                 |  "休息快结束啦"     | → 全面阻断 + AI 升级  |
```

**变更：**
- 服务器 policy 中 `overRest` 增加 `shouldTriggerActions` 字段
- iOS `blocking.service.ts` 改为检查 `shouldTriggerActions`（而非 `isOverRest`）
- 取消 XState 中 `ENTER_OVER_REST` 的 DB 写入

#### Sleep 双机制（在线 + 离线）

```
在线: Server → WebSocket → iOS App → blockingService → ManagedSettingsStore
离线: DeviceActivityMonitor App Extension → ManagedSettingsStore

两套写同一个 Store，互为备份，不冲突。
```

### 6.4 AI 行为教练系统

**利用已有的 AI Chat 能力 + 触发器框架，实现福格模型的三要素：**

#### 动机 (Motivation) — AI 连接当前任务与长期目标

| 时机 | AI 行为 | 示例 |
|------|---------|------|
| 晨间规划 | 连接今日任务与周目标 | "本周目标是完成 Auth 模块，今天如果做完登录页就完成 60% 了" |
| 番茄钟结束 | 展示进展 | "这个番茄钟你推进了 2 个小函数，Auth 模块整体进度 45% → 52%" |
| 超时休息 | 提醒目标 | "今天计划做 3 个番茄钟还差 1 个，如果现在开始 11:30 就能完成" |
| 晚间总结 | 回顾成就 | "今天完成了 5 个番茄钟，Auth 模块进度从 30% 到 65%，明天再加把劲" |

#### 能力 (Ability) — 降低行动摩擦

| 场景 | 摩擦点 | AI 简化方式 |
|------|--------|------------|
| 开始番茄钟 | 选任务+点开始 | AI 自动推荐"接下来做 XXX？"，用户点确认即启动 |
| 任务太大 | 不知道从哪开始 | AI 建议拆分："这个任务估计要 3 个番茄钟，建议先从 API 接口开始" |
| 被打断 | 只能中止 | AI 提供暂停选项："暂停 10 分钟？我帮你记住进度，回来继续" |
| Airlock | 必须选 3 个任务 | AI 根据优先级+截止日期自动建议 Top 3 |

#### 提示 (Prompt) — 在正确时机给出正确信号

| 触发条件 | 提示内容 | 渠道 |
|----------|---------|------|
| 进入 READY 后 3 分钟未开始 | "准备好了吗？上次你做 {task} 进展不错" | AI Chat |
| REST 剩余 1 分钟 | "休息快结束了，下一个想做 {suggestedTask} 吗？" | AI Chat + 通知 |
| 检测到在非黑名单娱乐网站停留 > 5 分钟 | "看起来你在浏览 {domain}，要不要开个番茄钟先？" | AI Chat (桌面端) |
| 连续 2 天没完成 Top 3 | "最近 {task} 一直没推进，要不要聊聊卡在哪了？" | AI Chat (晨间) |
| 番茄钟中切换了 3 次任务 | "今天切换了好几次任务，要不要固定一个先做完？" | AI Chat (REST 时) |

#### 触发器增强计划

在已有 `ai-trigger.service.ts` 框架上增加：

| 触发器 | 条件 | 动作 | 福格要素 |
|--------|------|------|---------|
| `idle_in_ready` | READY 状态停留 > 3 min | AI 推荐下一个任务 | 提示 + 能力 |
| `rest_ending` | REST 剩余 < 1 min | AI 建议下一个任务 | 提示 |
| `browsing_distraction` | 浏览器检测到非工作网站 > 5 min（非 FOCUS） | AI 温和提醒 | 提示 |
| `daily_plan_incomplete` | 连续 2 天 Top3 完成率 < 50% | AI 建议调整目标或拆分任务 | 能力 + 动机 |
| `morning_auto_plan` | 每日首次打开 App | AI 自动生成 Top 3 建议 | 能力 |
| `pomodoro_suggest` | READY 状态 + 有明确下一任务 | AI 主动问"开始吗？" | 提示 |
| `focus_protect` | FOCUS 中检测到频繁切 App | 不提醒（保护专注） | — |
| `weekly_pattern` | 周一晨间 | AI 总结上周模式，建议本周策略 | 动机 |
| `work_detected` | 检测到 IDE 活跃但无番茄钟 | AI 建议启动番茄钟或进入轻保护模式 | 提示 |
| `presentation_mode` | 检测到投屏/屏幕共享 | 自动暂停所有阻断和提醒 | — |

### 6.5 新能力：工作状态感知

**解决问题 P3（已经在工作但没番茄钟）**

```
信号源:
  桌面端: IDE 窗口活跃 (VS Code/JetBrains) → powerMonitor + frontmost app 检测
  桌面端: 键盘/鼠标活跃但非娱乐网站
  浏览器: 当前标签页在 GitHub/GitLab/Jira 等工作网站

判断逻辑:
  if (在工作时间 && 无番茄钟 && 检测到工作信号 > 5min) {
    AI: "看起来你已经在工作了，要不要开个番茄钟？
         这样我可以帮你记录时间，到时间提醒你休息。"
    // 或者自动进入"轻保护模式"——不开番茄钟，但阻断分心 App
  }
```

**轻保护模式（新概念）：**
- 不需要正式启动番茄钟
- 检测到用户在工作 → 自动激活分心应用阻断
- 检测到用户停止工作（切换到娱乐网站/空闲）→ 自动解除
- 没有严格的时间限制，适合不规律的工作节奏
- 可以在任何时候切换到正式番茄钟模式

### 6.6 新能力：展示/社交模式

**解决问题 P8（投屏尴尬）**

```
触发条件（自动检测）:
  macOS: CGDisplayStreamCreate 检测到屏幕共享
  macOS: 检测到 Zoom/腾讯会议/钉钉的屏幕共享进程
  手动: 用户对 AI 说"我要开会了"或点击"展示模式"按钮

行为:
  - 暂停所有弹窗、通知、阻断提醒
  - 浏览器扩展不再重定向/弹出 overlay
  - 桌面端不再置顶窗口
  - iOS 端不受影响（不需要，手机不投屏到会议）

恢复:
  - 自动: 屏幕共享结束后 → AI 问"会议结束了？要不要继续之前的任务？"
  - 手动: 用户说"会议结束了"或点击关闭展示模式
  - 超时: 默认 2 小时后自动恢复（防止忘记关闭）
```

### 6.7 新能力：AI 可协商的规则

**解决问题 P10（规则死板）**

**核心设计：所有规则都可以通过 AI 对话临时调整，但有约束。**

```
可协商的规则:
  ┌──────────────────────┬───────────────────────────┬──────────────┐
  │ 规则                  │ AI 可以做什么              │ 约束          │
  ├──────────────────────┼───────────────────────────┼──────────────┤
  │ 睡眠阻断时间          │ 推迟 1-3 小时             │ 不能取消当晚   │
  │ 今日番茄钟目标        │ 下调到 1 个               │ 不能设为 0     │
  │ 工作时间段            │ 临时缩短/延长             │ 不超过 14 小时 │
  │ 阻断暂停             │ 暂停 10-120 分钟          │ 自动恢复      │
  │ Top 3 任务           │ 替换/减少                 │ 至少 1 个     │
  └──────────────────────┴───────────────────────────┴──────────────┘

不可协商的规则（安全阀）:
  - 连续工作 > 2 小时必须提醒休息（健康保护）
  - FOCUS 中分心 App 阻断不能关闭（除非中止番茄钟）
  - 每日工作 > 10 小时发出强警告

AI 协商原则:
  1. 理解原因: "为什么需要调整？"
  2. 给出建议: "我可以这样帮你，但建议..."
  3. 设置补偿: "推迟睡眠 2 小时，但我帮你设一个 1 点的强制提醒"
  4. 记录模式: 如果用户频繁调整某个规则，建议永久修改默认值
```

**实现路径：**
- AI Chat 已有 28 个 MCP 工具，增加 `flow_adjust_sleep_time`、`flow_pause_blocking`、`flow_adjust_daily_goal` 等工具
- 高风险调整（取消当晚睡眠阻断）标记为需要确认的操作
- 所有调整记录在日志中，AI 在周报中总结："本周你调整了 3 次睡眠时间，平均推迟了 1.5 小时"

```
  ├── AI 根据用户状态动态调整番茄钟时长
  ├── 跨设备行为关联（桌面在 IDE → 手机不阻断工具类 App）
  └── 周/月维度的目标追踪与策略建议（连接 Goals/Projects）
```

### 6.9 批判性反思

以上设想需要保持清醒，有几个值得注意的风险：

**过度设计风险：** 轻保护模式、展示模式、AI 协商 — 每一个都增加系统复杂度。需要验证每个能力的真实使用频率，避免做了没人用。建议每个能力先做最小版本验证。

**AI 可靠性：** AI 判断不一定准确。检测到"用户在工作"可能是误判（只是打开了 IDE 但在发呆）。所有 AI 判断都应该是**建议**而不是**自动执行**，给用户确认/否决的机会。

**诚实前提：** AI 协商规则建立在"用户保持诚实"的前提上。如果用户每天都说"今晚有紧急 deadline"来推迟睡眠阻断，系统应该温和指出模式但不强制阻止——最终，这是用户自己的生活。

**番茄工作法的本质：** 番茄工作法的核心价值不是"25 分钟定时器"，而是**有意识的工作节奏**。轻保护模式虽然降低了摩擦，但也可能让用户失去"主动决定开始工作"这个仪式感。需要在便利性和仪式感之间找到平衡。

**简单可能是最好的：** 有时候用户不启动番茄钟，不是系统的问题，而是用户的选择。不是所有工作都适合番茄钟。系统应该**辅助**而不是**控制**。
